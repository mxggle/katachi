## 一、核心概念

### 1.1 什么是 Unit

Katachi 的最小学习单位叫 **Unit**，由三个维度组成：

- **单词** (wordId): 例如「食べる」
- **活用形** (conjugationType): 例如「て形」「丁寧形」「否定形」等
- **练习模式** (mode): 「选择题」或「输入题」

一个 Unit 的例子：`食べる × て形 × 选择题`

同一个单词的不同活用形是**独立**的 Unit，各自有独立的学习进度。

### 1.2 三种练习模式

| 模式 | 说明 |
|------|------|
| **每日练习 (daily)** | 主力模式。有每日目标、新词上限、SRS 调度 |
| **弱点训练 (weakness)** | 只从已学过的 Unit 中选题，不引入新词 |
| **自由练习 (free)** | 不受任何限制，从全部可用 Unit 中随机出题 |

### 1.3 每日目标体系

用户可以设置两个参数：

- **每日目标 (dailyQuestionGoal)**: 每天总共要练多少题，默认 20
- **每日新词上限 (dailyNewLimit)**: 每天最多引入多少个新 Unit，默认 5（自动计算为 dailyQuestionGoal × 25%）

## 二、SRS 间隔重复算法

### 2.1 Unit 的三种状态

每个 Unit 有一个状态机：

```
learning → review → graduated
```

| 状态 | 含义 | 触发条件 |
|------|------|---------|
| **learning** | 刚学的新词，需要短间隔巩固 | 初始状态；或答错时回退到此状态 |
| **review** | 正在复习中，间隔逐步拉长 | 从 learning 答对后进入 |
| **graduated** | 已掌握，长间隔维护性复习 | 复习间隔达到 21 天以上 |

### 2.2 间隔计算规则（当前实现）

**答对时：**

- 如果处于 `learning` 状态 → 直接跳到 `review`，间隔设为 1 天
- 如果处于 `review` 或 `graduated` → 新间隔 = 当前间隔 × ease 因子（向上取整）
- ease 因子 +0.1（最高 3.0）
- 如果新间隔 ≥ 21 天 → 状态变为 `graduated`

**答错时：**

- 无论当前什么状态 → 强制回到 `learning`，间隔归零
- ease 因子 -0.2（最低 1.3）

**下次复习时间的计算：**

- 间隔 = 0 时（learning 状态）：10 分钟后
- 间隔 > 0 时：当前时间 + 间隔 × 24 小时

### 2.3 Ease 因子

Ease 是一个乘数，决定间隔增长的速度：

- 初始值：2.5
- 答对一次：+0.1
- 答错一次：-0.2
- 范围：1.3 ~ 3.0

示例轨迹（全部答对）：

```
Day 0: learning → 答对 → interval=1天, ease=2.5
Day 1: review → 答对 → interval=3天(1×2.5≈3), ease=2.6
Day 4: review → 答对 → interval=8天(3×2.6≈8), ease=2.7
Day 12: review → 答对 → interval=22天(8×2.7≈22), ease=2.8 → graduated
Day 34: graduated → 答对 → interval=62天(22×2.8≈62), ease=2.9
```

## 三、每日练习的出题算法

### 3.1 出题流程概览

```
第一步：构建候选池（buildEligibleUnits）
   ↓
第二步：分池分类
   ↓
第三步：按优先级依次取题（selectFromPool）
   ↓
第四步：动词组轮询交叉（takeDiversifiedUnits）
```

### 3.2 第一步：构建候选池

从全部可用单词中，根据用户设置的 JLPT 等级、词性、活用形进行过滤。

**当前行为（存在问题）：** 对于同一个单词，只选择**1个活用形**进入候选池——选择规则是：优先已有进度的活用形，在同类中选最 overdue（nextReviewDate 最早）的那个。

这意味着如果「食べる」的「て形」和「丁寧形」同时到期，只有更紧急的那个会进入候选池，另一个被丢弃。

### 3.3 第二步：分池分类

所有已有进度的 Unit 被分为 5 个优先级池：

| 优先级 | 池名 | 定义 |
|--------|------|------|
| 1 | **mistakes (错题)** | 到期 + 最近答错过（consecutiveWrong > 0）或者处于 learning 状态且有错误记录 |
| 2 | **unstable (不稳定)** | 到期 + 正确率 < 80% + 不是 graduated 状态 |
| 3 | **dueReview (到期复习)** | 到期 + 不属于以上两类的正常复习项 |
| 4 | **new (新词)** | 没有任何进度记录的全新 Unit |
| 5 | **nonDue (未到期)** | 有进度但 nextReviewDate 还没到的 Unit（用作兜底填充） |

"到期"的定义：`nextReviewDate ≤ 当前时间`

### 3.4 第三步：按优先级取题

按照 1→2→3→4→5 的顺序依次从各池中取题，直到凑满 `questionCount`：

1. 先取所有 mistakes
2. 再取所有 unstable
3. 再取所有 dueReview
4. 取 new（上限为 dailyNewLimit 和剩余空位的较小值）
5. 最后用 nonDue 填满剩余空位

### 3.5 第四步：动词组轮询交叉

取题时不是简单地顺序取，而是按照动词组（godan 五段动词 / ichidan 一段动词 / irregular 不规则动词）做 **round-robin 轮询**：

```
示例：总共要取6题
→ godan词1 → ichidan词1 → irregular词1 → godan词2 → ichidan词2 → irregular词2
```

目的是避免连续出现同类型动词导致「模式匹配」，强制用户每道题都重新判断动词类型。

## 四、Session 机制

### 4.1 Session 生命周期

1. 用户点击「开始今日练习」
2. 系统检查今日已答题数，计算剩余 budget
3. 调用出题算法生成 session（题目列表 + 选项）
4. 用户逐题作答
5. 答错的题会被**放到队尾**重新出现（re-queue 机制）
6. 所有题目（包括重试的）全部答对后，session 结束

### 4.2 每日预算 (Daily Budget)

- 每日预算 = `dailyQuestionGoal - 今日已答题数`
- 如果预算 ≤ 0，每日练习模式会显示「今日已完成」
- 弱点训练和自由练习不受预算限制

### 4.3 答题记录

每次答题（包括重试）都会产生一条 attempt 记录，包含：
- 哪个 Unit
- 是否正确
- 答题时间

同时更新该 Unit 的进度：seen 次数、正确/错误次数、连续正确/错误次数、SRS 参数。

## 五、首页统计显示

首页「今日练习」区域显示 4 个数字：

| 项目 | 计算方式 |
|------|---------|
| 错误复习 | 满足 mistake 条件 + 已到期的 Unit 数量 |
| 不稳定 | 满足 unstable 条件 + 已到期的 Unit 数量 |
| 到期复习 | 普通到期的 Unit 数量 |
| 新题 | 完全没有进度记录的单词数量（上限为 dailyNewLimit） |

## 六、已知问题清单

### 🔴 严重问题

#### P1: 新用户首日体验断裂

**现象：** 新用户设置目标 20 题，但实际只能获得 5 题（因为 dailyNewLimit = 5，且没有任何复习项可填充）。

**原因：** `dailyNewLimit` 是硬上限，不考虑 review 池是否为空。当 review 池 = 0 且 nonDue 池 = 0 时，session 大小被强制限制在 dailyNewLimit。

**期望行为：** 新用户第一天应该能练满目标数量。随着复习积累，新词比例自然降低。

#### P2: 每词只选 1 个活用形

**现象：** 同一个单词的多个活用形同时到期时，只有最紧急的那个会被选中，其他被丢弃。

**原因：** 候选池构建阶段做了「每词取最优」的过滤。

**影响：** 用户的第二弱、第三弱活用形永远得不到练习，除非最弱的那个先 graduated。

#### P3: dailyNewLimit 不追踪跨 Session 的新词

**现象：** 用户做 2 次 daily session，每次都引入 dailyNewLimit 个新词。一天实际引入 2 × dailyNewLimit 个新词。

**原因：** scheduler 只在单次 session 内限制新词数量，不检查今天之前的 session 已经引入了多少新词。

### 🟡 中等问题

#### P4: 答错重试消耗每日预算

**现象：** 用户设置目标 20 题，但一道题答错 2 次再答对 = 消耗 3 次预算。答错率高的用户可能只练了 10 个不同的题就被告知「今日已完成」。

**原因：** 预算计算统计的是全部 attempt 记录数，包括 retry。

**影响：** 弱的用户练习量反而更少，与 SRS 目标矛盾。

#### P5: SRS 缺少学习阶段梯度

**现象：** 新词答对 1 次后，下次出现是 1 天后。没有 "10分钟后再练一次" 的短间隔巩固。

**原因：** learning 状态答对后直接跳到 review（interval = 1天），没有中间的学习步骤。

**影响：** 新词记忆巩固不足，用户在 session 中每个新词只见 1 次就消失了。

#### P6: Ease Hell（难度地狱）

**现象：** 某个词答错 6 次后，ease 降到 1.3（最低值）。之后即使连续答对，ease 每次只回升 0.1，需要 17 次连续答对才能回到 2.5。这个词会长期占据大量复习时间。

**原因：** 答错 ease -0.2，答对 ease +0.1，下降速度是上升速度的 2 倍。

#### P7: Graduated 答错惩罚过重

**现象：** 一个已经掌握的词（interval = 21天+），因为一次答错直接回到 learning 状态（interval = 0）。用户需要重新走完整个 learning → review → graduated 流程。

**原因：** 答错时不区分当前状态，统一回退到 learning。

#### P8: Non-Due 填充破坏 SRS 间隔

**现象：** 当复习项不够填满 session 时，系统会拿还没到期的 Unit 来凑数。

**影响：** 提前复习会破坏 SRS 的「遗忘曲线间隔」设计，让用户感觉「明明学过了怎么又来了」。

#### P9: 首页数字与实际 Session 不一致

**现象：** 首页显示「错误复习 3 + 不稳定 2 + 到期 5 + 新题 5 = 15」，但 session 可能只生成 10 题（因为 questionCount 限制）。

**影响：** 用户期望与实际不匹配，造成困惑。

## 七、参数默认值一览

| 参数 | 默认值 | 说明 |
|------|--------|------|
| dailyQuestionGoal | 20 | 每日总目标题数 |
| dailyNewLimit | 5 | 每日新词上限 (= goal × 25%) |
| questionCount | 10 | 单次 session 的题数 |
| 初始 ease | 2.5 | SRS 间隔倍率 |
| ease 上限 | 3.0 | — |
| ease 下限 | 1.3 | — |
| graduation 阈值 | interval ≥ 21天 | 达到后进入 graduated 状态 |
| learning 重试间隔 | 10 分钟 | interval=0 时的下次复习时间 |
